# 网关模块技术设计文档

## 1. 模块定位与职责

### 1.1 定位

网关（Gateway）是游戏服务的入口层，负责处理所有客户端连接，是客户端与游戏逻辑服务之间的唯一通信桥梁。

### 1.2 核心职责

| 职责 | 说明 |
|------|------|
| **连接管理** | 建立、维护、断开 WebSocket 连接，管理会话生命周期 |
| **消息路由** | 解析客户端消息，根据 MsgID 分发到对应的处理函数 |
| **心跳检测** | 定期检查客户端存活状态，清理超时连接 |
| **协议编解码** | 将原始二进制数据与业务数据包相互转换 |
| **并发控制** | 管理大量并发连接，保证服务稳定性 |

### 1.3 在系统中的位置

```
客户端(WebSocket) → Gateway(网关) → LoginService(登录) → GameService(游戏逻辑)
                                          ↓
                                    Redis(会话存储)
```

## 3. 网络协议设计

### 3.1 二进制协议格式

```
┌──────────────────────────────────────────────────────────────────┐
│                        消息包结构 (8 + BodyLen 字节)               │
├──────────────┬──────────────┬──────────────┬─────────────────────┤
│   MsgLen     │    MsgID     │    SeqID     │        Body         │
│   (4字节)    │   (2字节)    │   (2字节)    │   (MsgLen-8字节)    │
│   uint32     │   uint16     │   uint16     │      []byte         │
└──────────────┴──────────────┴──────────────┴─────────────────────┘
```

| 字段 | 大小 | 类型 | 说明 |
|------|------|------|------|
| `MsgLen` | 4 字节 | uint32 | 包总长度（含包头+包体） |
| `MsgID` | 2 字节 | uint16 | 消息类型 ID，用于路由 |
| `SeqID` | 2 字节 | uint16 | 消息序列号，用于请求-响应匹配 |
| `Body` | 变长 | []byte | 业务数据（Protobuf/JSON 序列化） |

### 3.2 粘包拆包方案

**方案：利用 WebSocket 帧机制**

WebSocket 协议本身保证了消息的边界性，每个 `ReadMessage()` 调用返回一条完整的消息，因此不需要额外处理 TCP 层的粘包问题。

```go
// session.go:104
_, data, err := s.conn.ReadMessage()  // WebSocket 保证消息完整性
reader := bufio.NewReader(&wsReader{data: data})
pkt, err := network.Decode(reader)    // 只解析本消息的内容
```

**编码流程**：

```go
// 发送方
data, err := network.Encode(msgID, seqID, body)  // 编码为二进制
conn.WriteMessage(websocket.BinaryMessage, data) // WebSocket 发送

// 接收方
_, data, err := conn.ReadMessage()               // WebSocket 接收完整消息
reader := bufio.NewReader(&wsReader{data: data}) // 包装为 io.Reader
pkt, err := network.Decode(reader)               // 解码为 Packet
```

### 3.3 消息 ID 定义

| MsgID | 消息类型 | 说明 |
|-------|----------|------|
| 1001 | HeartbeatReq | 心跳请求 |
| 1002 | HeartbeatResp | 心跳响应 |
| 1005 | ErrorResp | 错误响应 |

## 4. 核心设计

### 4.1 读写分离

每个 Session 启动两个独立 goroutine：

```go
// session.go:55-62
func (s *Session) Start() {
    go s.readLoop()   // 读 goroutine：接收客户端消息
    go s.writeLoop()  // 写 goroutine：发送消息给客户端
}
```

**读循环**：从 WebSocket 读取原始数据 → 解码 → 更新心跳 → 路由分发

```go
// session.go:97-159
for {
    _, data, err := s.conn.ReadMessage()  // 阻塞读取
    pkt, err := network.Decode(reader)     // 解码
    s.lastHeartbeat = time.Now()           // 更新心跳
    s.router.Handle(s, pkt)                // 路由处理
}
```

**写循环**：从 `writeCh` 取消息 → 编码 → 写入 WebSocket

```go
// session.go:161-206
for {
    select {
    case pkt, ok := <-s.writeCh:           // 从通道取消息
        data, err := network.Encode(...)   // 编码
        s.conn.WriteMessage(...)           // 写入连接
    case <-s.stopCh:                       // 停止信号
        return
    }
}
```

**优势**：
- 读操作不阻塞写操作，反之亦然
- 避免长时间业务处理阻塞消息接收
- 写通道缓冲 1024，非阻塞发送，防止慢客户端影响服务

### 4.2 连接管理

**生命周期**：

```
连接建立 → 创建 Session → 加入 Manager → 启动读写循环 → 心跳检测 → 连接断开 → 清理资源
    ↑                                                              ↓
    └───────────────────────────────────────────────────────────────┘
```

**Session 创建流程**（[server.go:166-209](file:///f:/Projects/go-snake-game/internal/gateway/server.go#L166-L209)）：

1. HTTP 请求升级为 WebSocket 连接
2. 创建 `Session` 封装连接
3. `SessionManager.AddSession()` 分配唯一 ID 并存入 `sync.Map`
4. `Session.Start()` 启动读写 goroutine
5. 阻塞等待 `stopCh`，会话结束后记录日志

### 4.3 中间件机制

**洋葱模型**：

```
请求 → MW1前置 → MW2前置 → Handler → MW2后置 → MW1后置 → 响应
```

**注册方式**：

```go
// router.go:38-39
r.Use(LogMiddleware)  // 默认挂载日志中间件
```

**中间件链构建**（[router.go:93-96](file:///f:/Projects/go-snake-game/internal/gateway/router.go#L93-L96)）：

```go
finalHandler := handler
for i := len(r.middlewares) - 1; i >= 0; i-- {
    finalHandler = r.middlewares[i](finalHandler)
}
```

**现有中间件**：

| 中间件 | 功能 |
|--------|------|
| `LogMiddleware` | 记录消息处理耗时，捕获 panic |

### 4.4 心跳超时机制

**检测流程**（[manager.go:164-233](file:///f:/Projects/go-snake-game/internal/gateway/manager.go#L164-L233)）：

1. `SessionManager` 启动后台 goroutine，每 10 秒扫描一次
2. 遍历所有会话，计算 `now - lastHeartbeat`
3. 超过 `heartbeatTimeout`（默认 60 秒）则调用 `session.Stop()`

**停止信号传播**：

```
checkHeartbeat() → session.Stop() → 标记离线 → close(stopCh) → close(writeCh) → conn.Close()
                                                    ↓                   ↓
                                              readLoop退出      writeLoop退出
```

## 5. 并发安全设计

### 5.1 sync.Map

用于存储所有在线会话，替代普通 `map`：

```go
// manager.go:28
sessions sync.Map
```

**特性**：
- 读操作完全无锁，性能极高
- 写操作只锁定特定 segment
- 即普通 map 的读写锁会让所有读操作排队，而 sync.Map 的读操作完全不排队
- `Range` 遍历时不会阻塞写操作
- 适合读多写少的场景（游戏网关典型模式）

### 5.2 sync.Once

**使用场景 1：Session.Stop() 防重复关闭**

```go
// session.go:68-94
func (s *Session) Stop() {
    s.closeOnce.Do(func() {
        // 标记离线、关闭通道、关闭连接、移除管理器
    })
}
```

**使用场景 2：全局管理器单例**

```go
// manager.go:52-61
func GetManager() *SessionManager {
    globalManagerOnce.Do(func() {
        globalManager = &SessionManager{...}
        globalManager.StartHeartbeatCheck()
    })
    return globalManager
}
```

### 5.3 单写 goroutine

WebSocket 的 `WriteMessage` 和 `Close` 方法必须互斥：

```go
// session.go:34
writeMu sync.Mutex  // 写锁

// session.go:186-189
s.writeMu.Lock()
err = s.conn.WriteMessage(websocket.BinaryMessage, data)
s.writeMu.Unlock()

// session.go:85-87
s.writeMu.Lock()
s.conn.Close()
s.writeMu.Unlock()
```

**原因**：`WriteMessage` 内部维护了写缓冲区状态，与 `Close` 并发调用会导致 panic。

### 5.4 通道关闭安全

```go
// manager.go:191-199
func (m *SessionManager) StopHeartbeatCheck() {
    select {
    case <-m.heartbeatStopCh:
        // 通道已关闭，避免重复关闭 panic
    default:
        close(m.heartbeatStopCh)
    }
}
```
